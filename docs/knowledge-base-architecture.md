## 一、背景与问题

### 1.1 初始状态

项目为 UiBot RPA "资金日报自动化"，使用 **UB 方言 (.task)** 和 **Python (.py)** 混合开发。原始知识库是一份 **467 行的 CLAUDE.md** 文件，包含了编码规范、UB 方言规则、API 速查表、数据库表结构、项目配置等全部内容，**没有任何分层或索引**。

### 1.2 核心痛点

| 痛点 | 具体表现 | 影响 |
|------|---------|------|
| 内容混杂 | 全局编码规范、UB 方言、项目专属表结构全部堆在一个文件 | 新项目无法复用，每次重写 |
| 无分层索引 | 467 行文件无目录、无标签、无导航 | AI 每次全文件扫描，平均消耗 200+ 行上下文 |
| 速查表低效 | 一份 45 行白名单大表，查一个函数要扫全表 | 查 "&" 要扫 45 行，找错函数时有发生 |
| 知识矛盾 | CDbl 在三处说法不一致（既说禁止又说可用） | 写代码时不知道该信哪个，产生 bug |
| 重复维护 | DB 约束在 2.3 和 2.4 两处各写一遍 | 修改时容易漏改，导致不一致 |
| 无外部 API 索引 | 官方 24 个模块 300+ 命令无法快速定位 | 每次冷门查询需盲搜整个 references 目录 |

### 1.3 目标

构建一个"检索快、命中准、可复制、可演进"的 AI 知识库系统：

| 指标 | 目标值 |
|------|--------|
| 热路径查询行数 | ≤ 60 行 |
| 冷路径首次查询行数 | ≤ 200 行 |
| 冷路径二次查询（行号缓存后） | ≤ 50 行 |
| 当前项目热命中率 | ≥ 90% |
| 知识库体积 | ≤ 50KB（可单文件夹复制） |
| 新项目部署时间 | ≤ 5 分钟 |

## 二、核心架构设计

### 2.1 三层缓存模型

借鉴 **CPU 多级缓存** 思想，将知识库分为三层：

| 层级 | 文件 | 数据量 | 加载时机 | 命中率 | 单次耗时 | CPU 类比 |
|------|------|--------|---------|--------|---------|---------|
| **L1 热** | CLAUDE.md (600行) | 55 个已验证命令 | 每次对话自动加载 | **95%** | 22~54 行 | L1 Cache (1ns) |
| **L2 冷** | uibot-api-full.md (300行) | 26 模块 174 命令 + 行号 | 按需跳转查阅 | 5% | 路由30行 + 精准跳15行 | L2 Cache (10ns) |
| **L3 源** | D:\\references\\ (24文件) | 全量函数签名 | L1/L2 都未命中时 | 0% (兜底) | 全文件 ~200 行 | 内存 (100ns) |

### 2.2 热知识库 — CLAUDE.md

CLAUDE.md 是核心索引，600 行，分为三大层：

- **第一章 [全局]**：跨项目通用编码规范，约 40 行
- **第二章 [领域]**：UiBot RPA 领域通用知识（UB 方言规则、4 张分类速查小表、Database/Excel/Dict/File API、Python 编码约束、容错模式模板），约 400 行
- **第三章 [项目]**：项目专属内容（DB 表结构、配置、自适应解析），约 130 行

### 2.3 冷路由表 — uibot-api-full.md

300 行的精准路由表，按 UiBot 官方 7 大类 26 模块组织。关键设计：
- 不重复抄写函数签名 —— **只做路由 + 行号索引**
- 每个命令标注：绝对路径、行号范围、用途、验证状态
- 行号缓存在文件中持久化，重启对话不丢失

### 2.4 三级检索流程

```
用户提问
  │
  ├─ L1 热: CLAUDE.md → 速查表定位 (~10 行)
  │   └─ 命中? → 直接返回
  │
  ├─ L2 冷: uibot-api-full.md → 路由到模块 (~30 行)
  │   └─ 有行号? → Read(Lxx-xx) → 精准返回 (~15 行)
  │
  └─ L3 源: D:\references\模块\文件.md → 读全文件
      └─ 自动写回行号到 uibot-api-full.md → 下次命中 L2
```

## 三、V1 → V6 演进路线

| 版本 | 做了什么 | 解决什么问题 |
|------|---------|------------|
| **V1** | 原始 CLAUDE.md 467 行 | 起点：全文件扫描，200 行/次 |
| **V2** | 三层分架 + 导航 + 标签索引 | 解决"找不到"，降到 40 行/次 |
| **V3** | CDbl 修复 + 白名单 1→4 + 去重 | 解决"找不准"和"维护不一致" |
| **V4** | 冷热分流 + 路由表 | 解决"冷门命令盲搜"，实现 100% 可达 |
| **V5** | D:\\references 去版本号 | 解决"跨项目跨机器不可复制" |
| **V6** | 行号缓存持久化 | 解决"冷查询每次重读全文件"，省 90% token |

## 四、关键技术细节

### 4.1 三重标签体系

每个知识点用标签标注其适用范围：

| 标签族 | 示例 | 含义 |
|--------|------|------|
| 全局层 | `[全局]`、`[全局→领域桥接]` | 跨项目通用 |
| 领域层 | `[领域·必读]`、`[领域·白名单]`、`[领域·黑名单]` | UiBot 领域通用 |
| 项目层 | `[项目·概述]`、`[项目·DB]`、`[项目·配置]` | 单项目专属 |
| 冷知识 | `[冷]`、`[冷·?]` | 按需查阅 |

### 4.2 分类小表设计

原 45 行一张白名单 → 拆分为 **4 张分类小表**（每表 4~15 行）：
- 字符串与类型转换 (14 行)
- 数组与集合 (4 行)
- 文件、Excel 与数据库 (15 行)
- 流程控制 (10 行)

AI 查一个函数只进一张表，扫表成本 **O(n/4)**。

### 4.3 行号缓存机制

最关键的优化。工作流程：

1. **首次命中**：查冷路由 → 无行号 → 读全文件 → 拿到答案
2. **写回缓存**：AI 自动将命令的行号范围写入 uibot-api-full.md
3. **二次命中**：路由表有 Lxx-xx → 只读 17 行 → **省 90% token**

行号持久化在文件里，**上下文压缩、重启对话、复制到新项目都不丢失**。

## 六、部署指南

### 前置条件（一次性）

将 UiBot 安装目录下的 `references/` 复制到 `D:\references\`（D 盘一级目录）。

```
D:\references\
├── AI/  基本命令/  鼠标键盘/
├── 界面操作/  软件自动化/
├── 系统操作/  网络/
└── index.md
```

### 知识库结构

```
knowledge-base/         ← 49KB，复制到任何项目
├── CLAUDE.md           # 热知识（全局 + 领域 + 项目专属）
├── uibot-api-full.md   # 冷路由（26 模块 + 行号索引）
├── settings.json       # 权限（D:\references\ 免弹框）
└── README.md           # 部署说明
```

### 部署步骤

1. 复制 `knowledge-base/` 到新项目根目录
2. `CLAUDE.md` + `uibot-api-full.md` 放项目根目录
3. `settings.json` 放 `.claude/` 目录
4. 修改 CLAUDE.md 第三章（项目专属层，5 分钟）

## 七、面试讲解提纲

### 30 秒电梯演讲

> "我把 AI 编程助手的知识库从一张 467 行全扫描的大表，重构为三级缓存架构。L1 热数据 55 个命令自动加载秒回，L2 冷路由 26 模块按需精准跳行，L3 官方源文件做行号缓存。单次查询从 200 行降到 40 行，命中率 95%，冷查询二次命中省 90% token。整个 49KB portable 包复制到新项目 5 分钟就能用。"

### 核心亮点

1. **冷热分层思想** — 借鉴 CPU L1/L2/L3 Cache，按"访问频率"做数据分级
2. **行号缓存机制** — 将 AI 的"短期记忆"（上下文）转为"长期记忆"（持久化文件）
3. **D:\\references 一级目录设计** — 路径从 105 字符 → 25 字符，真正 portable
4. **可量化效果** — 每个改动都有前后数据对比，用数字说话
